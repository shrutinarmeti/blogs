Building a CI/CD Pipeline with GitHub Actions

Automate your tests, linting, and deployments end-to-end — from pull request to production — without leaving GitHub.

Shipping software reliably is hard. Manual deployments are error-prone, and catching bugs after they reach production is expensive. A solid CI/CD pipeline changes this: every commit is automatically tested, every merge is automatically deployed, and your team gets fast feedback loops that make releasing fearless.

GitHub Actions lets you build these pipelines directly inside your repository using YAML workflow files. No external CI service required.

Core Concepts

  • Workflow — a YAML file in .github/workflows/ that defines when and what to run.
  • Event — the trigger for a workflow (push, pull_request, schedule, etc.).
  • Job — a set of steps that run on the same runner (VM).
  • Step — a single shell command or reusable Action.

A Simple CI Workflow

This workflow runs on every pull request targeting main, installs dependencies, lints, and tests a Node.js app:

name: CI

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Run tests
        run: npm test

Adding Continuous Deployment

After tests pass on main, automatically deploy to production. Here's a minimal example deploying a static site to GitHub Pages:

name: Deploy

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4

      - name: Build
        run: npm run build

      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist/

      - id: deployment
        uses: actions/deploy-pages@v4

Secrets and Environment Variables

Never hardcode credentials in workflow files. Store them in Settings → Secrets and variables → Actions and reference them as:

env:
  DATABASE_URL: ${{ secrets.DATABASE_URL }}
  API_KEY: ${{ secrets.API_KEY }}

Secrets are masked in logs automatically — GitHub redacts any log line that contains a secret value.

Caching Dependencies

Workflows that restore a cache on the second run are significantly faster. The actions/setup-node action handles npm caching automatically with cache: npm. For Python, use actions/setup-python with cache: pip. Manual caching via actions/cache gives you full control for other ecosystems.

Tips for Maintainable Workflows

  • Pin Actions to a full commit SHA (not a mutable tag) for reproducibility and supply-chain security.
  • Use reusable workflows (workflow_call) to share logic across repositories.
  • Add concurrency groups to cancel stale runs when a new commit is pushed.
  • Use matrix builds to test across multiple OS / runtime version combinations.
